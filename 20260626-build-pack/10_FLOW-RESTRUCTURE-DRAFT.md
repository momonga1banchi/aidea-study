# 10. デモとプレゼンの流れ再構成 草案

対象: `aidea-workshop-20260626/`

この文書は、2026-06-16時点の追加フィードバックを受けた修正方針案である。まだスライド、デモプロジェクト、講師資料には反映しない。

## 1. 今回の問題認識

現状の資料は、各デモの個別説明は成立しつつあるが、デモ同士の接続が弱い。

特に問題なのは次の点。

- Demo 1が「AIは間違える」という前提に依存している。
- 実際にはGPT-5.3 Codex Sparkのような強いエージェントは、テストがなくても重複箇所を見つけて正しく直す可能性がある。
- その場合、Demo 1の主張が崩れ、「間違えるAIを見せるデモ」に見えてしまう。
- Demo 2でcharacterization testを作っても、それが以後のSensorsやHarnessにどう接続するのかが見えにくい。
- Demo 3/4も「何ができるようになったか」は見えるが、「それでもリリース判断には何が足りないか」が明示されていない。
- 聴衆が「既存システムにAIで機能追加するとき、どの課題をどの順に潰していけばよいのか」を持ち帰りにくい。

したがって、中心テーマを次のように変更する。

> AIが間違えることを見せるのではなく、AIがうまく直せたとしても、それだけでは業務開発のリリース判断には足りないことを見せる。

## 2. 新しい中心メッセージ

現行の核は維持しつつ、デモの見せ方を次の一文へ寄せる。

> 既存システムにAIで変更を入れるときに必要なのは、「AIが直せるか」ではなく、「何を根拠にリリースしてよいと言えるか」を段階的に作ること。

このため、各デモは「AIの成否」ではなく、リリース判断に必要な材料がどこまで揃ったかを示す。

## 3. 達成レベルで全体を見せる

## 5. Demo 1の再定義

### 現行の問題

現行Demo 1は「AIがpricing.jsだけを直し、promotionService.jsを見落として矛盾する」筋書きに依存している。

しかし強いAIは次のように動く可能性がある。

- `rg "5000|FREE_SHIPPING|threshold"` 相当の調査を自発的に行う。
- `pricing.js` と `promotionService.js` の両方を直す。
- 簡単な手動確認まで行い、表面的には正しい結果を出す。

この場合、「AIが間違えるから危険」という見せ方は成立しない。

### 変更後の主張

Demo 1の主張を次に変更する。

> テストなしでもAIが正しく直すことはある。しかし、テストなしでは「正しく直ったことを継続的に証明できない」ため、そのままでは業務リリースの根拠にならない。

### Demo 1で見せる分岐

Demo 1は、AIの結果に応じてどちらでも同じ結論へ着地させる。

| AIの結果 | 見せるもの | 着地点 |
|---|---|---|
| 片方だけ直した | 矛盾curl、重複閾値、古いspec | 受け入れ根拠がないと欠陥が漏れる |
| 両方直した | diff、手動確認、古いspec、テスト不在 | 今回は直ったが、次回も守る根拠がない |

重要なのは、失敗を見せることではない。

「リリースしてよいですか？」と聞かれたときに、Demo 1の状態では次に答えられないことを見せる。

- 既存のどの挙動が変わっていないか。
- 境界値はどこまで確認したか。
- APIレスポンス形式は壊れていないか。
- 仕様書と実装の乖離はどう扱ったか。
- 次回の変更で同じ観点を再確認できるか。

### Demo 1後のステータス

| 項目 | 状態 |
|---|---|
| 機能変更 | できたかもしれない |
| 手動確認 | 一部できる |
| 回帰テスト | ない |
| 仕様との整合 | 不明 |
| 構造保証 | ない |
| 記録 | ない |
| リリース判断 | まだできない |

スライド13は「AIが間違えた証拠」ではなく、「テストなしでは証明できない」に改題するのがよい。

## 6. Demo 2の接続を明確にする

### 現行の問題

Demo 2でcharacterization testを作ること自体は良いが、作った後に「だから次に何ができるのか」が弱い。

### 変更後の主張

Demo 2の主張を次にする。

> まず既存挙動を固定すると、AIの変更後に「既存の何が壊れていないか」を機械で確認できる。

ただし、Demo 2だけではまだリリースできない。

### Demo 2で達成したこと

- 現在の5,000円境界を固定できた。
- 4999/5000の境界値を固定できた。
- APIレスポンス形式を固定できた。
- 送料とメッセージの不変条件を固定できた。
- 仕様書が古い場合に、docsではなく実挙動を根拠にできた。

### Demo 2でまだ足りないこと

- 7,000円へ変更する新仕様の受け入れ条件はまだない。
- AIがどのファイルを見て、どの制約を守るべきかはまだ伝えていない。
- テストは振る舞いを守るが、controllerへ業務ロジックを直書きするような構造劣化は止められない。
- 仕様変更後にspec / ADR / worklogをどう更新するかはまだ決まっていない。

### Demo 2からDemo 3へのつなぎ

言い換えると、Demo 2で手に入れたのは「振る舞いのセンサー」である。

次に必要なのは「構造のセンサー」。

ここでDemo 3へつなぐ。

> 振る舞いが壊れていないことは分かるようになった。では、動いているけれど設計を壊す変更はどう止めるか。

### スライド修正案

- Slide 14: 「既存挙動を固定する」に加え、「ここで作ったテストは以後のデモで最初から入っている」と明記する。
- Slide 15: 「テストがAIの足場になる」だけでなく、「足場になったが、まだ構造と変更依頼が足りない」を明示する。

## 7. Demo 3の接続を明確にする

### 現行の主張

Demo 3は「テスト緑でも構造違反を止める」という見せ方で成立している。

### 追加すべき主張

Demo 3後に、達成状態と不足状態を明示する。

### Demo 3で達成したこと

- テストが緑でも、controllerへの業務ロジック直書きを検出できる。
- 層の責務、依存方向、数値リテラルの増殖を機械チェックできる。
- 「動けばOK」ではなく「保守できる形か」をセンサーにできる。
- 既知負債はallowlistで許容し、新しい違反だけ止める現実的な運用を見せられる。

### Demo 3でまだ足りないこと

- 事前にAIへ作業範囲や調査原則を渡していない。
- 変更依頼の受け入れ条件が文書化されていない。
- センサーが赤になった後、AIがどう直し、何を記録すべきかがまだ決まっていない。
- 人間が最後に何を見るべきかがまだ整理されていない。

### Demo 3からDemo 4へのつなぎ

Demo 3までで、Sensorsは増えた。

次に必要なのは、作業前にAIへ渡すGuidesと、作業後に人間が判断するSteering。

> テストと構造センサーで「後から検出」はできるようになった。次は、AIに最初から何を守るべきか渡し、最後に人間が何を見るかを決める。

### スライド修正案

- Slide 16: test緑 / architecture赤の説明に加え、「L3: 構造センサー追加」とステータスを表示する。
- Slide 17: AGENTS.mdを単独概念として出すのではなく、「Demo 3までに足りなかったGuides」として出す。

## 8. Demo 4の位置づけを明確にする

### 現行の主張

Demo 4は、CR、AGENTS.md、テスト、センサーを揃えて同じ変更を行う山場として成立している。

### 強めるべき主張

Demo 4は「AIが正しく直した」ではなく、「リリース候補としてレビューできる状態になった」と見せる。

### Demo 4で達成したこと

- Change Requestにより、変更範囲と受け入れ条件が明示された。
- AGENTS.mdにより、AIへ調査原則、層の責務、記録ルールを事前に渡した。
- characterization testにより、既存挙動の破壊を検出できる。
- architecture / schema / lint / typecheckにより、振る舞い以外の劣化も検出できる。
- spec / ADR / worklogにより、判断と変更理由が次回の前提として残る。
- 人間は全コードを読むのではなく、CR、センサー結果、diff、記録を確認する役割に移れる。

### Demo 4でもまだ足りないこと

ワークショップ上は「リリース候補」まで到達するが、一般的な現場では次が残る。

- CIで同じセンサーを自動実行すること。
- 本番相当データや外部サービス連携の確認。
- セキュリティ、権限、監査ログ、性能など非機能のセンサー。
- 事業側による受け入れ条件の最終確認。
- デプロイ手順、ロールバック手順、監視項目。

ここを明示すると、話がワークショップ内の玩具に閉じず、プロの現場へ接続できる。

### Demo 4後のリリース判断

Slide 21または20で、次の表を出す。

| 確認者 | 見るもの | このデモでの状態 |
|---|---|---|
| 機械 | test / lint / typecheck / architecture / schema | green |
| AI | 赤の原因調査、修復、記録作成 | 実施済み |
| 人間 | CRとの整合、diff、残リスク、記録 | レビュー可能 |
| 現場CI | 同じチェックの自動実行 | 今後の導入課題 |
| 本番運用 | デプロイ、監視、ロールバック | スコープ外 |

### Demo 4からDemo 5へのつなぎ

Demo 4は、人間が途中で赤を見て、AIに修正を促す進め方。

Demo 5は、完了条件と停止条件をさらに機械可読にして、途中の往復を減らす進め方。

> Demo 4で「安全にレビューできる変更」は作れた。では、毎回人間が途中で見なくても、出口で判断できる形にできるか。

## 9. Demo 5の位置づけ

Demo 5は引き続きボーナス扱いでよい。

ただし、「全部任せるすごいデモ」ではなく、次のように位置づける。

> 完了条件と停止条件まで明文化すると、人間は入口と出口に集中できる。ただし、これはDemo 1-4の足場があるから成立する。

Demo 5で見せるべきこと。

- CRのfront-matterが、AIの残作業リストになる。
- `change-package` sensorが、必要成果物や期待挙動の不足を赤として出す。
- AIはloop-stateに赤と修復方針を記録する。
- completion-reportで、人間が読むべき情報を絞る。
- 人間の介入回数は減るが、人間の責任は消えない。

Demo 5後に必ず言うこと。

> 自律化は最初に目指すものではない。まずはDemo 2の既存挙動固定から始める。

参加者には、次の順序を持ち帰ってもらう。

1. AIは、テストがなくても正しく直せることがある。
2. しかし業務開発では、正しく直ったことを証明できないとリリースできない。
3. まず既存挙動をcharacterization testで固定する。
4. 次に、テストでは守れない構造をセンサーにする。
5. 変更依頼とAGENTS.mdで、AIへ事前条件を渡す。
6. センサーで機械確認し、人間はCR、diff、残リスク、記録を見る。
7. 判断をspec / ADR / worklogに残すと、次回のAI作業の前提になる。
8. 自律ループは最初に目指すものではなく、足場が揃った後に介入点を減らす発展形である。
