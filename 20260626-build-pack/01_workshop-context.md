# 01. 勉強会の背景と設計方針

## 開催概要

- 2026/06/26 開催、社内勉強会「AIdea勉強会」の初回テーマ回。全体60分、LIVE実演あり。
- オンライン+オフライン併催。録画あり。デモはハンズオンではなく発表者が見せる形式。

## テーマ構造

- **表向きのテーマ**: AIによるテスト自動化(参加者の関心テーマ。冒頭からこれで入る)
- **実際に伝える中心テーマ**: AIエージェントに安全に開発を任せるためのハーネスエンジニアリング入門
- 接続の論理: テスト自動化 → AIエージェントは実装もテストも書ける → しかし放置は危険(暴走リスク) → 安全に任せる足場=ハーネス。この順でブレイクダウンする。いきなりハーネスの話から入らない。

## 想定する聴衆

- SES現場で既存システムの保守・追加開発に関わるエンジニアが中心。
- AI開発に興味はあるが実務での使い方に迷っている人、テスト自動化に関心がある人、既存コードにテストが少なく変更が怖い人、AIに勝手なことをされるのが怖い人。非エンジニア寄りの参加者も一部いる。
- 理解度に幅がある前提でよい。「一人でも二人でも、これは使えそうと思う人が出ること」が成功条件。
- 熟練者向けに寄せすぎず、知らない人が「おっ」と思える内容にする。専門用語は避けすぎなくてよいが、初出時に一言で定義する。

## 持ち帰ってほしいこと(スライド冒頭2〜3枚目に必ず置く)

- AIエージェントは実装もテスト作成もかなりできるようになっている。
- ただし業務開発では「できました」だけでは足りない。AIの変更を安全に受け入れるには事前条件と事後チェックが必要。
- テストはAI開発における安全装置・足場になる。
- 既存システムにAIを入れるときは、まず既存挙動を固定するところから始める。

## プレゼン全体の核(一文)

> AIにコードを書かせること自体は、もう難しくない。
> これから重要になるのは、AIが書いたコードを安全に受け入れるための仕様・テスト・検証・記録の仕組みを作ること。

## 概念フレーム: ハーネスの3要素

デモとスライド全体をこのフレームで貫く。図で説明する(文字の羅列にしない)。

- **Guides / Feedforward**: 作業前にAIへ渡す前提情報・制約(仕様書、AGENTS.md、設計方針、ADR、禁止事項、変更範囲)
- **Sensors / Feedback**: 作業後に結果を確認するチェック(テスト、lint、typecheck、アーキテクチャチェック、スキーマ比較)
- **Steering**: 人間が判断し、その判断を記録して次回の前提にすること(レビュー、ADR更新、worklog、仕様書更新)

## 冒頭の問い(つかみ)

「テストがない既存システムで、AIに修正を頼めますか?」
具体例: 既存の注文金額計算APIに「5,000円以上で送料無料 → 7,000円以上に変更したい」という依頼が来た。
自分が関わっていないプロジェクトだったら、どこから見るか? AIに修正させたあと、何をもってOKと言えるか?

## 構成上の必須事項(資料レビューで確定済みの要求)

- 冒頭に具体的なケース・問いを置く(抽象論から入らない)。
- 2〜3ページ目に「今日持ち帰ってほしいこと」。
- デモの前に必ずデモ全体像(これから何を見るか)を1枚で示す。
- 1スライド1メッセージ。要所で「テスト自動化」のテーマに戻す。
- ADRやキャラクタライゼーションテストは必要以上に深掘りしない。
- 最後は「明日からできる小さな一歩」に落として締める。
- コード・ターミナル画面はフォントサイズをかなり大きめに。オンライン参加者に細かいコードを読ませない。「どのファイルか」より「何を確認しているか」を説明する。

## 制作上の禁止事項(再掲)

納品物(スライド・ガイド・README・デモ内ドキュメント)に、本資料群の制作履歴・バージョン番号・「以前の版からの改善点」を一切書かない。聴衆向けの物語は「現場の課題 → 解決アプローチ → デモ → 持ち帰り」だけで完結させる。
