# 06. 発表者向け解説資料の仕様

発表者は「この資料群を初めて見る人」と想定する。**facilitator-guide.mdとinstructor-notes.mdだけ読めば、内容を理解し当日実行できる**ことが合格条件。

## docs/workshop/ の構成

```
docs/workshop/
├── facilitator-guide.md        # 当日の進行台本(主資料)
├── concept-primer.md           # 発表者向け事前学習(15分で読める)
├── first-steps-checklist.md    # スライド26と同内容の配布用チェックリスト
└── presentation-outline.md     # スライド一覧と各スライドの意図(1行ずつ)
```

## facilitator-guide.md(150行以上。検収対象)

1. **全体進行表**: 01の時間割をそのまま転記し、各行にスライド番号とデモユニット名を対応付ける。
2. **セクションごとの台本**: 「言うこと(話し言葉のスクリプト)」「見せる画面(スライド番号/ターミナル/エディタ)」「切り替えタイミング」。
3. **デモ運用**:
   - 開始前チェックリスト(全デモで `create-fresh-run.sh` 実行→期待出力の確認)
   - LIVE実演の安全策(一時ディレクトリ・新規セッション・instructor-prompts/materialsを作業ディレクトリに入れない)
   - 失敗時フォールバック(各デモの録画/ログへの切り替え手順と、切り替え時の言い回し)
4. **時間調整**: 押した場合のスキップ順(05の圧縮版)、余った場合の追加トーク(allowlistの話、AGENTS.mdの書き方)。
5. **想定質問と答え(10問以上)**。最低限含めるもの:
   - 「テストをAIに書かせて、そのテストが間違っていたら?」(人間は仕様の代表値を確認する/不変条件のように値に依存しない検証を混ぜる)
   - 「AIがテストの方を書き換えて緑にしたら?」(既存テストの改変禁止をGuidesに書く+変更検知やCIで守る発展がある、と1分で答える)
   - 「うちはJava/PHPだが使えるか?」(ハーネスは言語非依存。センサーの実装が変わるだけ)
   - 「どのAIツールの話?」(特定ツールに依存しない。AGENTS.mdは主要エージェントが読む共通慣行)
   - 「導入は何から?」(スライド26の最初の2項目)
6. **機材・環境チェックリスト**: フォント拡大設定、ターミナルの配色、画面共有の事前テスト、録画、Meet URL、オフライン会場の見え方確認(6/23週にリハーサル)。

## concept-primer.md

発表者がハーネスエンジニアリングに詳しくない前提で、以下を平易に解説する:

- ハーネス/Guides/Sensors/Steeringの定義と、なぜこの順で説明すると伝わるか
- characterization test・ADR・AGENTS.mdとは何か(深掘りしすぎない程度の理解で十分、というガイド付き)
- デモ題材に仕込まれた2つの仕掛け(閾値の重複・古い仕様書)の意図と、それぞれがどのデモで発火するか
- このプレゼンが「テスト自動化」を入口に「安全に任せる仕組み」へ着地する論理構造

## 各demo-unit/instructor-notes.md

04の必須5項目(証明したいこと/開始前チェック/進行手順/結果と分岐/所要時間と省略点)。
コマンドはすべてコピー&ペーストで動く形で記載する。

## 禁止事項

発表者向け資料にも、本資料群の制作履歴・バージョン番号・「以前の版では」等を書かない。発表者に必要なのは当日の実行情報であって制作経緯ではない。
