## セクションごとの台本

### 1. オープニング(Slide 1-2)

#### 言うこと

今日は「AIにテストを書かせる方法」だけの話ではありません。AIが実装もテストも速く書くようになったとき、人間が何を見れば安心して受け入れられるのかを扱います。

AIdeaの開発現場でも、仕様、コード、テスト、運用判断が別々の場所にあります。AIはそれらを横断して作業できますが、横断できることと、正しく完了したと言えることは別です。

最初に期待値を合わせます。今日のゴールは、AIの能力紹介ではなく、AIの作業を安全に受け入れるための足場を実物で見ることです。

#### 見せる画面

- Slide 1でタイトルを見せる。
- Slide 2で勉強会の位置づけを説明する。
- ターミナルにはまだ切り替えない。

#### 戻り方

そのままSlide 3へ進む。

### 2. 持ち帰り・問い(Slide 3-5)

#### 言うこと

今日持ち帰ってほしいのは、AIは実装者にもテスターにもなるが、受け入れ責任者にはならないという感覚です。受け入れ責任を人間が持つなら、AIが出した結果を何で確認するかを先に決める必要があります。

問いは単純です。テストが薄い既存システムで、送料無料ラインを5,000円から7,000円へ変えてとAIに頼んだら、何を見てOKと言えるでしょうか。

この問いを、あとで5つのデモに分けて確認します。最初はわざとハーネスなしで失敗を見せます。

### 3. 考え方と題材(Slide 6-11)

#### 言うこと

テスト自動化は、単にテストコードを書く作業を自動化することではありません。AIが変更したあとに、人間が判断する材料を自動で集めることです。

そのための足場を、今日はハーネスと呼びます。GuidesはAIに渡す前提、Sensorsは機械的な確認、Steeringは人間の判断を次の前提へ戻すことです。

題材は小さな注文見積りAPIです。controller、service、repository、configに分かれています。罠は2つあります。送料無料閾値が2箇所に重複していること、仕様書が3,000円のまま古いことです。

### 4. Demo 1: ハーネスなし(Slide 12-13)

#### 言うこと

最初は、あえて何も足場を置きません。AIに1文だけ渡すと、もっとも近い定数だけを直して「できました」と返ってくることがあります。

ここで大事なのは、AIが悪いという話ではありません。人間側が確認方法を渡していないので、AIはどこまで確認すべきかを判断できません。

矛盾レスポンスを見ます。送料は500円のままなのに、メッセージは送料無料です。この状態でも、テストがなければ緑も赤もありません。

#### 見せる画面

- Slide 12でrun-demo.mdの1文を見せる。
- ターミナルに切り替える。

```bash
cd demo-units/01_no_harness/project
perl -0pi -e 's/FREE_SHIPPING_THRESHOLD = 5000/FREE_SHIPPING_THRESHOLD = 7000/' src/config/pricing.js
HOST=127.0.0.1 PORT=31101 node src/server.js
curl -sS -X POST http://127.0.0.1:31101/orders/estimate \
  -H 'content-type: application/json' \
  -d '{"items":[{"sku":"UNIT-001","quantity":6000}]}'
```

LIVEでは直接ファイルを汚さず、fresh-runのコピーで実行する。詰まったら `demo-units/01_no_harness/instructor-materials/contradiction-curl.txt` を表示する。

#### 戻り方

Slide 13へ戻る。レスポンス中の `shippingFee\":500` と `message\":\"送料無料\"` を指差してから次へ進む。

### 5. Demo 2: 挙動固定(Slide 14-15)

#### 言うこと

次にAIへいきなり実装させず、今の振る舞いを固定させます。仕様書は古いかもしれないので、コードと実レスポンスを根拠にします。

ここで作りたいのは、正しい未来のテストではなく、現在地を固定するテストです。境界値、スキーマ、不変条件が入っていれば、次の変更で何が壊れたか分かります。

3分以上かかる場合はcompleted-testsへ切り替えます。LIVEを成功させることより、どんな観点を固定したかを説明するほうが重要です。

#### 見せる画面

```bash
rg "3,000|5000|FREE_SHIPPING" docs src

node --test
```

### 7. Demo 4: ハーネスあり(Slide 18-21)

#### 言うこと

ここが山場です。同じ送料無料変更でも、CR、AGENTS.md、テスト、センサーがある状態で頼みます。期待する筋書きは、赤、全文検索、重複発見、緑です。

AIが一発で直しても問題ありません。その場合はログを見せて、なぜOKと言えるかを確認します。重要なのは、AIが賢かったかどうかではなく、受け入れ条件を機械で確認できることです。

赤テストでは、送料計算とメッセージ計算が別の閾値を見ていることが表に出ます。全緑sensors表では、テスト、lint、typecheck、architecture、schemaが揃って戻ったことを確認します。

### 8. ボーナス・まとめ(Slide 22-28)

#### 言うこと

最後は、入口と出口だけ人間が見る形に近づけます。front-matter付きCRには、期待挙動、必要成果物、停止条件が入っています。

自律ループの価値は、AIが勝手に作業することではありません。赤になった理由をloop-stateに残し、完了時にはcompletion-reportで人間が見るべき点を絞ることです。

まとめます。AIに任せるのではなく、AIに安全に任せられる状態を作る。まずは代表的な入出力を3つ固定するところから始めれば十分です。

#### 見せる画面

## 想定質問と答え

### Q1. テストをAIに書かせて、そのテストが間違っていたら?

人間は代表値と不変条件を確認します。AIが書いたテストをそのまま信じるのではなく、仕様の観点を人間が設計します。

### Q2. AIがテストの方を書き換えて緑にしたら?

既存テストの改変禁止をGuidesに書き、CIや変更検知で守ります。今日は入口だけ触れます。

### Q3. うちはJava/PHPだが使えるか?

使えます。変わるのはセンサーの実装で、Guides / Sensors / Steering の考え方は同じです。

### Q4. どのAIツールの話?

特定ツールには依存しません。AGENTS.mdのようにリポジトリへ前提を書く慣行が重要です。

### Q5. 導入は何から始めるべき?

代表的な入出力をメモし、AIに既存挙動テストを作らせるところから始めます。

### Q6. 既存仕様書が古いときはどうする?

コードと実レスポンスで現状を復元し、乖離はworklogへ残します。

### Q7. センサーを増やすほど開発が遅くならないか?

最初は軽いセンサーだけで十分です。重要なのは人間の確認を減らす順に育てることです。

### Q8. 小さい案件でもADRは必要?

大げさな文書ではなく、判断理由を1行残すだけでも次回の前提になります。

### Q9. 非エンジニアは何を見ればよい?

緑か赤か、何が確認されたか、何を人間が判断するかを見ます。

### Q10. CIがない現場ではどうする?

まずローカルのnpm scriptから始めます。CIは後で権威として移します。

### Q11. AIにどこまで任せるべき?

ハーネスが整っている範囲までです。完了条件と停止条件が書ける作業から始めます。

### Q12. 失敗したデモをどう扱う?

ログへ切り替えて、なぜ失敗したか自体を教材にします。
