# Demo 1: テストなしでAIに修正を頼む

## このデモで証明したいこと

AIはテストがなくても正しく修正できることがある。問題はAIの成否ではなく、テストや記録がない状態では「リリースしてよい」と言える根拠が残らないことである。

## 開始前チェック

fresh-runで作ったコピーだけを開く。元の `project/` は汚さない。

```bash
cd project
rg "5000|FREE_SHIPPING|threshold" src docs
```

開始時点では閾値が `src/config/pricing.js` と `src/services/promotionService.js` に存在し、仕様書には古い3,000円が残っている。

## 進行手順

1. `instructor-prompts/run-demo.md` の1文だけをAIに渡す。
2. AIが直した差分を確認する。両方の閾値を直しても、片方だけでも、話の着地点は同じにする。
3. 片方だけの修正だった場合は、矛盾curlを見せる。
4. 両方直った場合は、差分と手動確認を見せたあと、テスト不在と古い仕様書へ話を切り替える。
5. 最後に「この状態でリリース承認できますか」と問い、足りないものを表で確認する。

## 片方だけ直した場合の見せ方

```bash
HOST=127.0.0.1 PORT=31101 node src/server.js
curl -sS -X POST http://127.0.0.1:31101/orders/estimate \
  -H 'content-type: application/json' \
  -d '{"items":[{"sku":"UNIT-001","quantity":6000}]}'
```

`shippingFee: 500` と `message: "送料無料"` が同時に出る。ここでは「AIが弱い」という言い方にしない。確認する足場がないと、欠陥が人間の手作業でしか見つからないと説明する。

## AIが重複まで直した場合

その場合は、成功を否定しない。次の切り返しを使う。

```bash
rg "3,000|5000|7000|FREE_SHIPPING" docs src
```

話す要点は次の通り。

- 今回は正しく直ったように見える。
- しかし、境界値、レスポンス形式、既存挙動、仕様書との乖離を次回も同じように確認する仕組みがない。
- 手動curlは証拠ではなく、その場限りの確認である。

## 達成したこと

- AIに機能変更を依頼できた。
- 実装差分や手動curlで一部の挙動を確認できた。
- テストなしの状態では、確認が人間の作業に戻ることを示せた。

## まだ足りないこと

- 既存挙動を守る回帰テストがない。
- APIレスポンス形式を継続確認できない。
- 仕様書が古いままでも気づく仕組みがない。
- 構造違反や責務逸脱を止めるセンサーがない。
- 判断理由と残リスクが記録されない。
- そのため、Demo 1の状態ではリリース判断はできない。

## フォールバック

片方だけ修正した失敗分岐は `instructor-materials/contradiction-curl.txt` を表示する。AIが正しく直した場合も、このログは「もし確認が漏れると何が起きるか」の補助資料として使う。
