# Demo 2: 既存挙動をcharacterization testで固定する

## このデモで証明したいこと

Demo 1では、AIが正しく直せたとしても受け入れ根拠が弱かった。Demo 2では、まず現在の挙動をテストに変換し、以後の変更で「何が壊れていないか」を機械が答えられる状態にする。

## 開始前チェック

```bash
cd project
rg "3,000|5000|FREE_SHIPPING" docs src
```

仕様書は3,000円、コードは5,000円で食い違っている。AIには、古いdocsだけでなくコードと実レスポンスを根拠にさせる。

## 進行手順

1. `run-demo.md` をAIに渡し、現在の注文金額計算APIの挙動調査を依頼する。
2. いきなりテストを書かせず、公開インターフェース、境界値、不変条件、エラーケース、docs差分の観点表を先に出させる。
3. 境界値4999/5000、APIレスポンス形式、不正入力、送料と残額の不変条件が入っているかを見る。
4. 生成が3分を超えたら、完成済みテストへ切り替える。
5. `node --test` を実行し、緑になったことを確認する。
6. 「ここで作ったテストは、Demo 3とDemo 4の初期状態に入っている」と明示する。

## completed-testsへの差し替え

```bash
mkdir -p project/tests
cp instructor-materials/completed-tests/*.js project/tests/
cd project
node --test
```

## 達成したこと

- 人間が貼るプロンプトで、観点表とテスト生成の順番を指定できた。
- 既存の5,000円境界をテストで固定できた。
- 4999/5000の境界値を繰り返し確認できる。
- APIレスポンス形式をテストで守れる。
- 送料、残額、メッセージの整合を不変条件として確認できる。
- 仕様書が古くても、実挙動を根拠に現在地を固定できる。
- 複雑なシステムでも、全体を網羅するのではなく、変更対象に近い公開契約、境界値、不変条件、代表ケースから押さえる方針を示せる。

## まだ足りないこと

- ここで生成したのは主に `tests/` であり、構造センサーや作業規約はまだ生成していない。
- 7,000円へ変える新仕様の受け入れ条件はまだない。
- AIに作業範囲や調査原則を渡していない。
- 送料無料判定が複数箇所へ分裂するような構造劣化は、振る舞いテストだけでは止められない。
- 仕様変更後にspec、ADR、worklogを更新する約束がまだない。

## 仕様変更後の扱い

characterization testは、既存挙動を永久に正しいものとして残すテストではない。7,000円へ変更するときは、次の3つに分ける。

| 種類 | 扱い | 今回の例 |
|---|---|---|
| 変えてはいけない挙動 | 回帰テストとして残す | response形式、不正入力、total計算 |
| 意図して変える挙動 | 新仕様の受け入れテストへ書き換える | 4999/5000境界を6999/7000境界へ変更 |
| 古い挙動の記録 | ADRやworklogへ移す | docsは古かった、閾値が重複していた |

言い方は「characterization testは変更前の地図です。変更後は、残すもの、書き換えるもの、記録へ移すものに分けます」とする。

## 古い仕様書への言及がなかった場合

「docs/specs/order-estimate.md と実レスポンスは一致していますか」とAIに聞く。ここで仕様書を直すのではなく、既存挙動固定の根拠をどこに置くかを確認する。
