# 講師用進行ガイド：日本語版

## 目的

この勉強会では、Codex Desktop / Claude Desktopを使い、AIに開発作業を進めてもらう方法を実演する。

ただし、主題はツール紹介ではない。  
主題は、AIに安全に作業させるためのハーネスをどう用意するかである。

## 伝えたい中心メッセージ

AIコーディングの品質は、モデルの賢さだけではなく、人間が用意したハーネスで決まる。

ここでいうハーネスとは、以下を指す。

- 仕様
- 制約
- テスト
- lint
- 差分レビュー
- 完了条件
- 禁止事項
- 作業範囲

## デモの基本方針

悪い例と良い例で、AIに見える情報を完全に分ける。

- 悪い例：仕様書・AGENTS.md・CLAUDE.md・タスクファイルなし
- 良い例：仕様書・AGENTS.md / CLAUDE.md・タスクファイルあり

Codex / Claudeで開くのは、必ず各デモ単位の `project/` のみ。  
`instructor-prompts/` は講師だけが見る。

## 最初からあるAPI

各プロジェクトには以下が存在する。

```text
GET /health
GET /tax?price=1000
```

この既存APIを使って、プロジェクトの構成を参加者に説明する。

## 追加する改修

```text
GET /discount?amount=50000
```

会員ランクと割引率を返すAPIを追加する。

## 推奨進行

### 1. 悪い例

AIに以下だけを伝える。

```text
会員ランクごとの割引率を返すAPIを追加してください。
日本語で説明してください。
```

観察ポイント：

- 仕様を勝手に決めていないか
- テストを書いているか
- 境界値を考えているか
- 既存のレスポンス形式に合わせているか
- controller / service の責務分離ができているか

参加者への問い：

```text
この差分をそのままPRにできますか？
何が不安ですか？
人間がレビューするとしたら、どこを見ますか？
```

### 2. 良い例

AIにタスクファイルを1つずつ読ませる。

```text
tasks/01-inspect-and-plan.md を読んで、その指示だけ実行してください。
まだファイルは編集せず、計画を報告した時点で止めてください。
回答は日本語でお願いします。
```

その後、以下の順で進める。

1. `tasks/01-inspect-and-plan.md`
2. `tasks/02-add-tests-only.md`
3. `tasks/03-implement-minimum.md`
4. `tasks/04-review-diff.md`

## 最後のまとめ

CodexとClaudeのどちらが上か、という話ではない。

重要なのは、AIにコードを書かせる前に、仕様・制約・テスト・完了条件・レビュー観点を渡すこと。

AIに丸投げするのではなく、AIが安全に作業できる環境を人間が作る。  
これがハーネスエンジニアリングである。
